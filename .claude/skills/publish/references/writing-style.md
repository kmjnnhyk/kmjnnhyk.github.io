# 글 작성 스타일 가이드

## 참조

사용자의 Medium 블로그: https://medium.com/@kmjnnhyk

## 문체 특징

### 톤
- **명사형/`~음`·`~됨` 종결의 보고체**. 사용자의 개인 말투(반말 캐주얼)를 모방하지 않음.
- 예시:
  - `~를 하던 도중 ~한 문제가 발견됨.`
  - `이번 작업을 통해 ~ 교훈을 얻음.`
  - `고민을 통해 ~하는 방향으로 작업을 결정하게 됨.`
  - `~한 점이 문제로 부상.`
  - `~을 ~로 정리.`
  - `~를 채택.` / `~로 결론.`
- 단조로운 반복을 피하기 위해 명사구 종결(`~`. )과 `~음`·`~됨`을 섞어서 사용.
- 1인칭 표현(`나`, `내가`)·자기 독백·감정 표현(`;;`)·의문형 도입부 모두 사용하지 않음.

### 구조
- 문제 → 분석 → 결정 → 결과의 흐름. 단계별 소제목으로 분리.
- 도입부는 맥락만 한두 문장. "이 글에서는 ~"식 메타 도입 금지.
- 마무리는 결과·교훈·후속 과제 한 문단. "결론적으로", "요약하자면" 같은 정형 마무리 금지.

### 제목
- **명사구 + `~기` 접미사**의 주제 중심 형식.
  - 예: `네이티브 로그 수집을 통한 앱 디버깅 및 성능 분석기`
  - 예: `AI 협업 환경의 마일스톤 및 워크플로우 재구조화기`
- "~한 회차", "~한 이야기" 같은 보고체·일기체 어미 사용 금지.
- **구체 숫자·고유명·날짜 제외**. "5대", "60분", "Samsung Galaxy" 같은 구체 디테일은 본문에서만 사용. 제목은 한 단계 추상화된 도메인·주제 명사로.
- **제목 무게는 실제 작업 규모와 균형 맞춰야 함**. 거창하게 부풀리지 말 것.
  - 거창한 명사(`파이프라인 설계`, `아키텍처 구축`, `체계 재구조화`, `시스템 재설계`)는 실제로 그 규모의 작업(여러 모듈 통합·장기 설계·구조적 결정 다수)일 때만 사용.
  - 측정 유틸 추가, 옵션 변경, 단일 함수 작성, 단발성 디버깅 등 작은 작업은 가벼운 명사(`도입`, `추가`, `정리`, `셋업`, `폐기`, `대체`, `점검`)로 표현.
  - 판단 기준: "이 제목만 본 사람이 상상하는 작업 규모와, 본문에서 확인되는 실제 규모가 일치하는가". 일치하지 않으면 과대 포장.
- 발표 주제·책 제목 톤. 본문 내용을 보지 않은 사람이 도메인을 짐작할 수 있어야 함.

### 설명 방식
- 기술 용어는 그대로 사용. 비유로 풀지 않음.
- 코드 스니펫보다 개념·판단 근거·트레이드오프 중심.
- 어떤 옵션을 검토했고 왜 어느 쪽을 택했는지 명시.

### 추상화 수준 — 글의 목적은 서사
- 글의 목적은 **"어떤 문제·고민이 있었고, 어떻게 해결해나갔고, 결과는 어떻게 되었는지"의 서사**. 구현 일지가 아님.
- 구체 구현 시나리오·함수명·변수명·특정 화면 단위 디테일은 한 단계 추상화하여 표현.
  - ❌ `login → terms modal 미표시` (구체 화면·시나리오)
  - ✅ `인증 상태 전환과 후속 UI 표시의 타이밍 충돌` (문제의 종류)
  - ❌ `useAuth.setAuthorized() 추가` (코드 변경 일지)
  - ✅ `진입 권한 부여 시점을 명시적으로 분리` (의미)
- 비슷한 결의 디테일이 여러 개라면 **하나의 카테고리로 묶어서** 한 섹션으로. 시도한 우회 방안들은 한두 줄로 압축.
- 판단 기준: "이 문장이 본문 서사를 진행시키는가, 아니면 단순 구현 회상에 그치는가". 후자는 삭제 또는 한 줄로 압축.

### 금지 사항
- **경어체** — `~입니다`, `~하겠습니다`, `~할 것입니다`.
- **`~다` 평서문** — `~했다`, `~한다`, `~이다`. (이번 톤은 `~음`·`~됨`·명사구 종결로 통일.)
- **AI 발표 톤** — `먼저 ~을 살펴보겠습니다`, `이 글에서는 ~을 알아보겠습니다`.
- **과잉 부사·빈 수식어** — `효과적으로`, `체계적으로`, `강력하고 유연한 도구`.
- **사용자 개인 말투 모방** — 사용자의 Medium 톤(반말 캐주얼·자기 독백 등) 따라하지 않음.
