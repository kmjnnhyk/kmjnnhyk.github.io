프로덕션에서 60분 오디오 세션이 끝난 뒤 세션 종료 API가 Network Error로 실패하거나, 세션 조기 종료 시 종료 호출 자체가 일어나지 않는 사례가 누적됨. 이전 회차에서 두 가지 실패 패턴(정상 종료 후 API 실패 / 조기 종료) 및 토큰 갱신 race condition을 식별. 이번 회차의 목적은 "왜 이런 실패가 구조적으로 반복되는가"를 진단하고, 그에 대응하는 아키텍처 방향을 좁히는 것.

## 문제 진단

근본 원인은 단일 문장으로 압축됨 — **세션 데이터의 영속화가 실시간 네트워크 전송에 100% 의존**.

이 의존이 만들어내는 실패 경로는 다음과 같음.

- OS suspend / kill — JS 스레드가 멈추면 진행 중이던 네트워크 호출도 같이 정지. `try/catch`로 잡히지 않는 "정지" 상태.
- 네트워크 차단 — 배터리 정책·App Standby·Doze 등으로 DNS·소켓이 막히면 요청은 pending에서 시간이 얼어붙음.
- 토큰 만료 — 세션 종료 직전 토큰이 만료되면 재발급 race가 겹쳐 종료 알림이 누락.

세 경로 모두 **네트워크 가용성 가정이 깨지는 순간 데이터 자체가 유실**됨. 이걸 클라이언트 레벨 try/catch나 retry로 막는 데에는 구조적 한계가 있음.

## 후보 가설별 검증

### 가설 1. SQLite를 도입하면 OS suspend 중에도 저장 가능

**부분 기각.** SQLite도 결국 앱 프로세스 안의 코드가 실행되어야 동작함. OS가 프로세스를 suspend하면 SQLite도 같이 멈춤.

다만 의미가 살아남는 지점이 있음 — **suspend 직전에 로컬에 확실히 기록**하고, 이후 sync하는 패턴. 네트워크 호출(수백 ms)과 달리 SQLite INSERT(0.01 ms)는 OS가 프로세스를 suspend하기 전에 완료될 확률이 압도적으로 높음. "OS suspend 중 저장"이 아니라 "suspend 전에 디스크 도달"이 핵심.

### 가설 2. catch 기반 fallback (정상 시 서버 직접 호출, 실패 시에만 SQLite 저장)

**기각.** OS가 프로세스를 kill하거나 JS 스레드를 suspend하면 네트워크 요청 중이던 코드의 catch 블록 자체에 도달이 불가능. fallback이 실행될 기회 없이 데이터 유실.

또한 catch에서 SQLite 저장을 시도하더라도, 그 시점이 OS의 cleanup 직전이라 SQLite write가 끝까지 도달한다는 보장이 없음.

### 가설 3. Write-ahead 패턴 (SQLite 선기록 → 서버 전송 → synced 마킹)

**채택.** 데이터가 발생한 즉시 SQLite에 INSERT하고, 그 이후에 서버 전송을 시도하며, 성공 시 `synced=1`로 마킹. 실패하거나 OS kill되더라도 데이터는 이미 디스크에 있음.

정상 동작 시 오버헤드는 SQLite INSERT(0.01 ms) 한 번이 추가되는 정도라 네트워크 호출(200~500 ms) 대비 1/10000 수준으로 무시 가능. 비정상 상황의 데이터 보존 가치가 정상 상황의 미세 오버헤드를 압도.

## 자료구조 후보 비교

| 후보 | 평가 |
|---|---|
| AsyncStorage | key-value 구조라 조건 검색 불가. `synced=0` 같은 부분 수정 시 전체 덮어쓰기 필요. 동시성 보장 부족. |
| JSON 파일 | atomic write 불가. 동시성 보장 없음. |
| SQLite | ACID 트랜잭션 / 조건 검색 / OS 내장 / JSI 기반으로 native 호출 오버헤드 거의 0. 채택. |

## react-query와의 관계 정리

세션 데이터 쓰기는 이미 react-query를 사용하지 않고 직접 호출하는 구조였음. 종료 알림만 mutation으로 감싸진 상태. SQLite는 react-query를 대체하는 것이 아니라, **쓰기 호출의 앞단에 로컬 영속화 레이어를 끼우는 것**으로 정리. 읽기 쿼리(`useQuery`)는 변경 없음.

## 백엔드 스키마 복제 여부

**불필요.** 로컬 SQLite는 서버 DB의 미러가 아니라, **서버로 보낼 API 요청을 재현할 수 있는 최소 필드만 보관하는 전송 큐**. 서버 스키마와 독립적으로 단순한 형태로 유지.

## sync 트리거 시점 결정

`AppState 'active'` 단순 트리거는 부적합. 로그아웃 상태에서 재개될 경우 401로 sync 자체가 실패함. 토큰 유효성이 보장되는 3개 지점에서만 sync를 트리거하도록 결정.

| 트리거 | 의미 |
|---|---|
| 홈 화면 진입 | 일반적인 foreground 복귀 경로. 로그인 상태에서만 도달. |
| 세션 종료 모달 표시 | 세션 직후 가장 빠른 전송 기회. |
| 로그인 성공 | 토큰 만료로 로그아웃 후 재로그인된 케이스의 복구 경로. |

## 영향 범위

- 신규 모듈 — 데이터베이스 초기화 / 세션 리포지토리 / sync 서비스.
- 변경 파일 — 세션 데이터 쓰기 경로의 호출부, 종료 알림 호출부, 로그인 성공 처리부.
- 백엔드 — 동일 session id로의 중복 호출에 대한 멱등성 확인이 선행되어야 함.
- 기존 플래그 의미 변경 — "서버 전송 성공" → "SQLite 저장 성공"으로 의미 이전. 호출부에서 사실상 항상 true.

## 결과

이번 회차에서는 아키텍처 방향과 sync 트리거 전략까지 확정. 구현 자체는 팀 리뷰 이후로 미룸. 외부 공유용 제안서를 작성하여 설계 근거(가설 기각 근거 포함)를 함께 정리.

## 교훈

- 아키텍처 변천이 사용자의 질문 한 줄("정상 시 매번 SQLite 거치는 건 낭비 아닌가?", "OS kill 시 catch에 도달 못 하지 않나?", "로그아웃 상태면?")에 의해 단계적으로 정제됨. 초안에서 도출된 설계가 그 자체로 최종 답이 아니며, **반례를 향한 질문을 누가 던지느냐**가 설계 품질을 결정.
- 클라이언트 영속화는 "OS가 프로세스를 정지시키기 전에 디스크에 쓰는 시간 경쟁"으로 다시 정의될 수 있음. 이 관점이 잡히면 SQLite 도입의 의미가 "오프라인 지원"이 아니라 **"네트워크 호출의 동기 의존을 제거"**로 명료해짐.
- sync 트리거는 단순 라이프사이클 이벤트가 아니라 **인증 상태가 보장되는 지점**에 묶여야 함. 트리거 후보를 식별하는 단계에서 "토큰이 유효한 것이 보장되는가"를 1차 필터로 사용.
