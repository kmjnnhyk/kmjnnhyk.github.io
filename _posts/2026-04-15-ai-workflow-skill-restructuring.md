사이드 프로젝트 하나를 굴리는 동안 마일스톤·의사결정 스킬·지속 문서가 동시에 정돈되지 않은 상태였음. 이번 회차의 목표는 마일스톤 범위 자체를 다시 좁히고, 그 위에 작업 진행 체계와 AI 협업 스킬 구조를 함께 재설계하는 것.

## 문제 진단

- M1 마일스톤의 범위가 "운영 전반"으로 잡혀 있어 베타 검증 타겟이 불명확.
- 제품 의사결정 스킬이 비전·로드맵 외에도 Phase 분리·구현 순서·기술 선택까지 끌어안고 있어 실행 영역과 책임이 충돌.
- 독립 스킬(DB 규칙, RN 성능 가이드, 디자인 가이드, 스킬 빌더, 워크플로우 가이드, git commit, git PR 등)이 다수 존재. 상위 스킬과 영역이 겹치거나 외부 플러그인과 중복되어 실제 발동 빈도가 낮음.
- `vision`·`roadmap` 지속 문서가 빈약하여 의사결정 근거가 휘발되는 구조.

## 마일스톤 재정의

기존 M1의 "운영 전반"을 **스태프 영역만**으로 축소. 게스트 영역은 채널사 의존성과 외부 변수 비중이 높아 베타 검증 비용이 큰 영역으로 판단됨. 통제 변수가 적은 스태프 측을 먼저 검증하고, 베타 5곳의 피드백을 기반으로 M2 이후 방향을 결정하는 순서로 변경. 게스트 관련 기능은 일괄 후순위 마일스톤으로 이관.

이 결정의 핵심은 "범위를 좁혀 베타 시그널의 해상도를 높임"에 있음.

## 의사결정 스킬의 책임 경계 재정립

제품 의사결정 스킬의 책임 범위를 **비전·로드맵·마일스톤 수준**으로 한정. Phase 분리·구현 순서·기술 선택은 실행 스킬로 분리. `SKILL.md`에 ✅ / ❌ 목록을 명시하여 경계를 박제.

이 정리를 통해 의사결정 스킬의 본문이 가벼워지고 호출 시 컨텍스트 점유가 감소. 실행 스킬과의 우선순위 충돌도 해소됨.

## 스킬 계층 정리

- **상위 스킬의 references로 흡수** — DB 규칙·RN 성능 가이드·디자인 가이드를 백엔드/프론트엔드/퍼블리셔 스킬 내부 references로 이동. `SKILL.md` 본문은 얇게 유지하고, 깊은 내용이 필요할 때만 references 로드.
- **외부 플러그인과 중복되는 스킬 제거** — git-commit, git-pull-request, workflow-guide, skill-creator 등은 외부 플러그인이 더 잘 다루는 영역이므로 자체 버전 폐기.
- **리뷰 전용 스킬 → 비전·로드맵 보유 스킬로 재정의** — 단순 리뷰 도구가 아니라 제품 의사결정의 단일 출처가 되도록 역할 확장.

## 지속 문서 보강

- `vision.md` 재작성 — 비전, 문제 정의, 대상 사용자, 앱 구조, 차별점, 수익 모델을 단일 문서로 통합.
- `roadmap.md` 재작성 — 마일스톤별 전략 목표·성공 지표 포함.
- `references/history/` 신설 — 굵직한 리뷰 결정 3건(기획문서 전체 리뷰, M2~M4 순서·수익 모델 재설계, M1 최소 MVP 재정의)을 박제하여 의사결정의 배경을 재추적 가능한 단일 출처로 확보.

## 결과

- 순삭감 약 2,000줄. 절반 이상이 문서 정리.
- M1 범위가 좁혀져 다음 액션의 우선순위가 명료해짐.
- 의사결정 스킬 1곳만 보면 비전·로드맵·리뷰 히스토리까지 추적 가능한 구조로 일원화.
- 상위 스킬은 `SKILL.md`를 얇게 유지하고 references는 필요 시 로드하는 패턴으로 호출 비용을 통제.

## 교훈

AI와 협업하는 워크플로우의 정돈은 도구를 늘리는 방향이 아닌, **결정의 출처를 좁히고 책임 경계를 명확히 하는 방향**으로 진행되어야 함. 스킬 수가 많을수록 호출 우선순위가 모호해지고, 비대해진 스킬은 컨텍스트 비용을 증가시킴. 마일스톤·책임 경계·지속 문서 세 축을 동시에 정리해야 이후 작업 진행이 안정되는 것으로 확인됨.
