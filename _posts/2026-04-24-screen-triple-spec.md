스크린이라는 단위를 어떻게 다룰 것인가에 대한 규약을 점진적으로 만든 시리즈의 마무리 회차. 출발점은 라우터 구조 재설계였지만, 작업 도중 "스크린 단위로 의도와 코드와 테스트가 정합을 이뤄야 한다"는 더 큰 문제가 부각되었고, 이후 회차들이 그 방향으로 이어짐. 이 글은 라우터 재설계를 작업 배경으로 두고, 거기서 도출된 스크린 트리플(`.tsx` / `.md` / `.test.tsx`) 규약의 설계 과정과 최종 형태를 정리.

## 작업 배경 — 라우터 구조 재설계

회차의 시작점에서 라우터 영역에 5가지 문제가 누적되어 있었음.

| 항목 | 증상 |
|---|---|
| 파일명 컨벤션 혼재 | 케이싱이 섞여 있음 |
| 레이아웃 파일 과다 | 트리 전반에 다수 분포 |
| 라우팅이 effect 안에서 처리됨 | 인증 분기가 부수효과화되어 흐름 추적이 어려움 |
| 의미 모호한 폴더 다수 | URL과 파일 트리의 의미가 어긋남 |
| 인라인 헤더 컴포넌트 | 화면별 헤더가 재사용 불가 |

방향은 인증 분기를 라우팅 레이어에서 **선언적**으로 처리하는 구조 변경. 구현 과정에서 다음 종류의 함정이 단계별로 드러남.

- **라우트 그룹의 URL 충돌** — 도메인 폴더까지 group 폴더로 만들면 URL 경로에서 그룹이 제거되며 동일 경로를 가진 화면들이 충돌. 인증 분기에만 group 폴더를 사용하고 도메인 폴더는 일반 폴더로 유지하는 방향으로 확정.
- **컴포넌트 재마운트와 hooks 일관성** — guard 값 변경 시 컴포넌트가 재마운트되며, 외부 store에 대한 hook 사용량이 마운트 전후로 달라지면 React가 hooks 정합성을 깨뜨림. component-level hook 호출을 끊고 store 직접 접근 형태로 전환하여 회피.
- **인증 상태 전환과 UI 전환의 타이밍 충돌** — 인증 상태를 단일 boolean으로 표현하자, 한 값이 바뀌는 순간 라우팅·UI·후속 모달이 동시에 흔들림. 시각적 가림이나 외부 storage flag로는 마운트 부작용까지 막지 못함이 확인됨. 결국 **인증 상태를 `로그인 됨` × `진입 권한 있음`의 두 축으로 분리**하여, UI 전환과 라우팅 전환이 서로 다른 단계에 놓이도록 만들어 해결.

라우터 재설계 자체는 완료되었지만, 이 과정에서 더 큰 문제가 부각됨.

## 본 작업으로의 전환

라우터 재설계 후에도 **개별 스크린이 무엇을 하는 단위인지가 코드만으로는 추적되지 않는다**는 문제가 남아 있었음. 한 스크린의 핸들러 동작 규칙, 변경 시 영향 범위, 테스트가 어떤 케이스를 커버하는지 — 이런 정보들이 코드 곳곳에 흩어져 있어 변경 영향 추적 비용이 높았음. 라우팅 레이어를 정돈해도 스크린 내부의 정합성 문제가 해소되지 않음을 확인.

이 시점에서 회차의 중심이 라우터에서 **스크린 단위의 정합 규약**으로 옮겨감.

## 명세 규약 도입 — 스크린 sibling `.md`

스크린 코드 파일 옆에 동일 이름의 `.md`를 두어 **UI 요소·핸들러 명세**를 기록하는 규약을 도입.

라우팅 측면 검증 — 빌드 도구가 `.md`를 라우트로 등록하지 않음을 스모크 테스트로 확인. 이 검증을 거쳐 sibling 배치가 안전함을 확정.

핸들러 명세 포맷 — 한 줄 요약 + params/testID/Trigger + Cases. Cases는 `조건 / 액션 / 결과`의 세 컬럼으로 표현하되, **요구사항(자연어)** 과 **설계(테스트로 옮겨질 프리미티브)** 두 층을 병기. 자연어는 비개발자 협업용, 프리미티브는 테스트 코드와의 1:1 매핑용.

스크린 코드 파일에는 핸들러 위에 `@see ./<screen>.md#<slug>` 형태의 pointer JSDoc만 두어 명세 중복을 제거.

## 첫 적용 — 인증 화면

가장 먼저 트리플로 구축된 스크린. 여기서 다음 패턴들이 확립됨.

- 테스트의 `describe(<handler>)` → `it(Case N. <요약>)` 1:1 매핑.
- mock factory를 별도 디렉토리에 두고, 테스트 파일에서 `createXxxMock()`을 가져와 주입하는 방식.
- 테스트 디렉토리는 라우터의 파일 기반 라우팅과 충돌하지 않도록 라우터 트리 외부에 배치.

이 사례가 이후 모든 스크린 트리플의 템플릿이 됨.

## 에지 케이스 — 라이프사이클·리다이렉트 스크린

초기 규약은 암묵적으로 **사용자 인터랙션 중심 스크린**을 가정. 하지만 다음 두 종류의 스크린에는 그 포맷이 맞지 않음.

- **라이프사이클 스크린** — 레이아웃 단위처럼 effect 기반 동작이 본체인 경우. 사용자 인터랙션 핸들러가 없거나 부수적.
- **리다이렉트 스크린** — 화면 자체가 조건에 따라 다른 경로로 보내는 브릿지인 경우. UI가 사실상 없음.

또한 기존 규약은 이 둘을 "생략 대상"으로 박아 두어, 실제 코드에 있는 의사결정이 어디에도 명세되지 않은 채 남는 문제가 있었음.

해결은 두 갈래.

1. **포맷 분화** — 스크린 유형을 3종으로 정의하고, 유형별로 필수 섹션을 다르게 둠.
2. **훅 리팩토링** — 훅 안에서 자체 effect로 동작하던 lifecycle wiring을 호출부의 effect로 꺼내, 명세 대상에 노출. 코드 양은 늘어나지만, "이 effect는 어떤 deps에서 어떤 동작을 트리거하는가"가 명세에 그대로 드러남.

훅의 shape 변경은 트레이드오프 — 코드 양 증가를 감수하더라도, **명세의 Trigger와 effect deps가 1:1로 일치**하도록 만드는 것이 변경 영향 추적의 비용을 더 크게 줄인다는 판단.

## 최종 규약 — 스크린 트리플

```
<screen>.tsx           코드 (handler + pointer JSDoc)
<screen>.md            명세 (3 유형 중 택 1)
<screen>.test.tsx      테스트 (Case 1:1 매핑)
```

스크린 유형은 3종.

| 유형 | 필수 섹션 |
|---|---|
| Interaction-driven | UI 요소 + Handlers |
| Lifecycle-driven | Derived State + UI 요소 + Handlers (lifecycle 네이밍) |
| Redirect bridge | Redirect Rules (서브포맷) |

핵심 불변식.

- 명세의 Case 설계 = 테스트의 `it()` 블록과 1:1.
- 명세의 Trigger = `useEffect` deps 와 1:1.
- 코드 파일의 핸들러 위 JSDoc은 명세 pointer만. 명세 본문을 코드에 중복 기재하지 않음.
- mock factory는 별도 디렉토리에 일관 배치. 테스트는 factory 호출로 주입.

## 결과

- 라우터 재설계 — 인증 분기가 라우팅 레이어에서 선언적으로 처리됨.
- 스크린 트리플 규약 확립 — 코드·명세·테스트가 같은 단위로 묶임.
- 인증 화면·라이프사이클 스크린·리다이렉트 스크린 모두 트리플 형태로 구축.
- 훅 shape 변경 — lifecycle wiring이 호출부 effect로 노출됨.
- 테스트 전체 통과.

## 교훈

- 라우팅 레이어 정돈만으로는 변경 영향 추적이 충분히 단순해지지 않음. **스크린의 의도를 명세화**해야 코드·테스트의 정합이 유지됨.
- 명세 형식은 도구의 라우팅 등록 정책과 충돌하지 않아야 함. 도입 전 라우팅 측면 검증을 선행할 것.
- 명세는 자연어와 프리미티브의 두 층으로 작성. 자연어는 비개발자 협업, 프리미티브는 테스트 코드 1:1 매핑.
- 스크린은 단일 카테고리가 아님. 사용자 인터랙션·라이프사이클·리다이렉트 등 본질적으로 다른 종류가 존재하므로, **명세 포맷은 유형별로 분화**되어야 함.
- 훅이 자체 effect로 모든 wiring을 흡수하면 명세 대상이 줄어드는 대가로 추적성을 잃음. 호출부에 effect를 노출하는 비용이 명세 정합성의 가치를 상회.
