도구의 자동 성능 모니터링(트레이싱, frames tracking 등)은 의도하지 않은 데이터가 과다하게 쌓이는 경향이 있고, 사용자 조작 패턴에 의존한 측정은 신호 대 잡음 비율이 낮음. 이번 회차의 목적은 **관심 있는 특정 시점의 FPS·메모리만 정확히 캡처**하고, 임계 초과 시에만 알림이 발생하는 파이프라인을 설계하는 것. 측정 도입의 동기와 검토했던 측정 방법들을 정리.

## 측정 도입 동기

- 자동 트레이싱을 활성화한 결과, 모든 인터랙션에서 trace 이벤트가 발화 — 어느 시점의 수치가 의미 있는지 사후 분리가 어려움.
- `enableUserInteractionTracing`, `reactNavigationIntegration` 등을 추가해 봤으나 신호 대 잡음 비율 개선이 미미하여 폐기.
- 화면 마운트, 모드 전환과 같은 **명확한 트리거 시점**에서의 수치만 보고 싶다는 요구가 분명하게 잡힘.
- 성능 저하 발생 시 slack으로 즉시 알림이 와야 하며, 알림에는 어떤 이벤트에서 저하됐는지 식별 가능해야 함.

## 운영 환경 트러블슈팅

본격 측정에 앞서 환경 세팅 단계에서 두 가지 트러블슈팅이 선행됨.

- **environment 필터 미스매치** — DEV 빌드에서도 environment가 `production`으로 기본 설정되어 있어 Issues에 이벤트가 노출되지 않음. DEV/prod 분리 후 정상 수신 확인.
- **level 별 inbound filter** — `level: 'info'`로 보낸 이벤트가 보이지 않음. 도구 측 inbound filter에서 info 레벨이 필터링될 수 있음을 확인. 검증 단계에서는 `level: 'error'`로 임시 강제하여 파이프라인 동작을 먼저 확보.

도구 도입 초기에는 측정 자체보다 운영 세팅 검증이 큰 비중을 차지함이 확인됨.

## 데이터 적재 방식 후보 검증

| 후보 | 평가 |
|---|---|
| Metrics API + 커스텀 태그 group by | SDK 버전에서 커스텀 태그가 attributes에 포함되지 않음 → 기각 |
| 메트릭 이름을 이벤트별로 분리 | 대시보드의 Bar Categorical에서 단일 메트릭만 선택 가능 → 다이벤트 비교 불가 → 기각 |
| Logs 데이터셋 + 커스텀 속성 | 커스텀 속성 기반 group by 가능. `perf_event` 속성을 키로 사용 → 채택 |

## FPS 측정 방식 결정

- **전역 `requestAnimationFrame` 루프** — 상시 CPU 점유가 발생. 측정 자체가 측정 대상의 성능을 깎는 역효과 가능. 기각.
- **온디맨드 측정** — 호출 시점부터 30프레임만 수집 후 정지. 캡처 1회당 약 500 ms 측정 지연이 발생하지만 상시 부담은 0. 채택.

메모리 수치는 별도 한 번의 호출로 가져오므로 추가 부담 없음.

## 알림 임계값 판단 위치

| 구성 | 평가 |
|---|---|
| 로그만 사용 | 대시보드는 정상 동작. Alert에서는 모든 로그의 평균만 사용 가능 → 어느 이벤트에서 저하됐는지 식별 불가. |
| 이슈만 사용 | Slack 알림에 이벤트명을 실을 수 있음. 다만 대시보드에서 숫자 집계 불가(태그는 문자열). |
| **로그 + 이슈 병행 (채택)** | 모든 측정은 항상 로그로 전송하여 대시보드 누적. 임계 초과 시에만 추가로 이슈를 전송하여 Slack 알림. 이중 전송이 발생하지만 위험 상황은 드물어 성능 영향 미미. |

임계값 판단을 외부 도구의 Alert 룰에 위임하지 않고 **앱 안에서 판단**한 이유는, 알림 메시지에 이벤트명을 직접 실을 수 있기 때문. 외부 도구의 룰만으로는 평균값 기반 판단이라 어느 시점이 문제였는지 추적이 어려움.

## 모듈 분리

- **측정 유틸** — `measureFPS`, `getMemoryMB` 등 순수 측정 함수.
- **전송 레이어** — 로그 전송, 이슈 전송 함수. 도구 SDK 의존을 한 곳에 격리.
- **성능 캡처** — 측정 + 임계 비교 + 전송을 한 호출로 묶은 함수와 React 훅.

모듈 경계가 명확해지면서 화면 코드는 캡처 트리거(이벤트명·임계값) 만 명시하면 되는 형태로 단순해짐.

## 영향 범위

- 정상 상황 — 로그 1회 비동기 전송. UI 블로킹 없음.
- 위험 상황 — 추가로 이슈 1회 전송. 드문 케이스이므로 누적 부담 미미.
- 전역 `requestAnimationFrame` 루프 제거로 상시 CPU 사용 0.
- 캡처 호출 시점에 약 500 ms 측정 지연 발생.

## 결과

- 측정 유틸·전송 레이어·캡처 훅을 별도 모듈로 분리하여 재사용 가능 형태로 정리.
- 대상 화면들에 측정 코드를 일괄 부착하여 동작 검증 후 다시 제거. 유틸만 남겨 두고 필요 시 부착하는 형태로 운영.
- 대시보드 — Logs 데이터셋, X축은 `perf_event`, 시각화는 `avg(fps)`.
- 알림 — Issue Alert 룰에서 메시지에 `[perf-alert]`가 포함된 경우 Slack 채널로 발송.

## 교훈

- "자동으로 다 측정"은 분석 단계의 신호 분리 비용을 사용자에게 떠넘기는 결과로 이어짐. **무엇을 측정할지를 트리거 시점부터 명시**하는 편이 데이터 가치를 높임.
- 임계값 판단은 "알림에 어떤 정보를 실어야 하는가"에 따라 위치가 결정됨. 외부 도구의 Alert 룰만으로는 이벤트 단위 식별이 어려운 경우가 많음.
- 도구 도입 초기 단계에서는 측정 설계보다 environment·inbound filter 같은 운영 세팅 검증에 더 큰 시간 비중이 들어감. 파이프라인 안정화 이전에 측정 결과를 신뢰하지 않을 것.
